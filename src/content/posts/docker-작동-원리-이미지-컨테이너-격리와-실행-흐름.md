# 한 줄 요약

Docker는 애플리케이션과 실행에 필요한 파일·라이브러리·설정을 이미지로 묶고, 호스트 운영체제의 커널을 공유하는 격리된 프로세스(컨테이너)로 실행하는 플랫폼이다. 가상 머신처럼 운영체제 전체를 부팅하지 않으므로 시작이 빠르고 자원 사용량이 작다.

# 1. 왜 Docker가 필요한가

개발 환경에서 ‘내 컴퓨터에서는 되는데 서버에서는 안 된다’는 문제는 런타임 버전, 시스템 패키지, 설정 파일, 환경 변수 차이에서 발생한다. Docker는 실행 환경을 코드로 선언해 개발·테스트·운영에서 같은 이미지를 사용하게 만든다.

# 2. 핵심 구성요소

- Dockerfile: 이미지를 만드는 레시피다. 베이스 이미지, 파일 복사, 패키지 설치, 실행 명령을 단계별로 적는다.

- Image: Dockerfile을 빌드해 만든 읽기 전용 레이어 묶음이다. 이미지 태그로 버전을 식별한다.

- Container: 이미지에 쓰기 가능한 컨테이너 레이어와 실행 설정을 더해 실제로 실행 중인 인스턴스다.

- Registry: 이미지를 저장·배포하는 저장소다. Docker Hub, GitHub Container Registry 등이 있다.

- Docker Engine: 클라이언트 명령을 받아 이미지·컨테이너·네트워크·볼륨을 관리하는 데몬이다.

# 3. 실행 흐름

1. docker build가 Dockerfile의 각 명령을 실행해 레이어 캐시를 가진 이미지를 만든다.

1. docker run이 로컬 이미지가 없으면 레지스트리에서 pull한다.

1. Engine이 이미지 레이어 위에 컨테이너 전용 writable layer를 만든다.

1. Linux 커널 기능으로 프로세스·파일시스템·네트워크·사용자 리소스를 격리한다.

1. ENTRYPOINT 또는 CMD가 지정한 프로세스가 PID 1로 시작된다.

1. PID 1 프로세스가 종료되면 컨테이너도 종료된다. 컨테이너는 작은 VM이 아니라 격리된 프로세스라는 점이 핵심이다.

# 4. VM과 컨테이너의 차이

VM은 하이퍼바이저 위에서 게스트 OS와 커널을 각각 실행한다. 강한 격리 대신 부팅과 메모리 비용이 크다.

컨테이너는 호스트 커널을 공유하며 사용자 공간과 프로세스만 분리한다. 따라서 수 초 안에 시작하고 같은 서버에 더 많이 배치할 수 있다. 단, 커널을 공유하므로 이미지 신뢰성, 권한, 네트워크 정책을 더 신경 써야 한다.

# 5. 격리는 어떻게 동작하는가

- namespaces: 프로세스가 보는 세계를 분리한다. PID namespace는 프로세스 목록, mount namespace는 마운트 지점, network namespace는 인터페이스·IP·포트, user namespace는 사용자 ID를 나눈다.

- cgroups: CPU, 메모리, I/O, 프로세스 수 같은 자원 사용량을 제한·회계 처리한다.

- union filesystem: 여러 읽기 전용 이미지 레이어와 하나의 쓰기 레이어를 겹쳐 하나의 파일시스템처럼 보이게 한다. 변경된 파일만 컨테이너 레이어에 기록된다.

- capabilities와 seccomp: root 권한을 세분화하고 위험한 시스템 콜을 제한해 공격 표면을 줄인다.

# 6. 이미지 레이어와 캐시

Dockerfile의 RUN, COPY, ADD 같은 명령은 대체로 새 레이어를 만든다. 앞 단계가 바뀌지 않으면 기존 레이어를 재사용하므로 빌드가 빨라진다.

예를 들어 의존성 파일을 먼저 COPY하고 패키지를 설치한 뒤 애플리케이션 소스를 COPY하면, 소스만 수정했을 때 패키지 설치 레이어를 캐시에서 재사용할 수 있다.

# 7. 네트워크와 데이터

컨테이너는 기본 bridge 네트워크에서 독립 IP를 받는다. -p 8080:80은 호스트 8080 포트와 컨테이너 80 포트를 연결한다. 같은 사용자 정의 네트워크의 컨테이너는 서비스 이름으로 서로 찾을 수 있다.

컨테이너의 writable layer는 컨테이너 삭제 시 사라진다. DB 데이터나 업로드 파일은 Docker volume 또는 bind mount에 둬야 지속된다.

# 8. 간단한 예제

Dockerfile 예시:

FROM node:22-alpine

WORKDIR /app

COPY package*.json ./

RUN npm ci --omit=dev

COPY . .

EXPOSE 3000

CMD [node, server.js]

실행 명령:

docker build -t hello-docker:1.0 .

docker run --rm -p 3000:3000 --name hello hello-docker:1.0

첫 명령은 재현 가능한 이미지 hello-docker:1.0을 만들고, 두 번째 명령은 이를 컨테이너 프로세스로 실행한다. --rm은 종료된 컨테이너를 자동 삭제하고, -p는 포트를 공개한다.

# 9. 실무 체크리스트

- 작은 신뢰 가능한 베이스 이미지를 사용하고 이미지 태그를 고정한다.

- 멀티 스테이지 빌드로 빌드 도구를 최종 이미지에서 제외한다.

- 컨테이너를 root가 아닌 사용자로 실행한다.

- 비밀값은 이미지에 COPY하지 말고 환경 변수·Secret 관리 도구로 주입한다.

- 상태 데이터는 volume 또는 외부 DB에 둔다.

- 헬스 체크, 로그 수집, 리소스 제한을 운영 환경에 설정한다.

# 기억할 점

이미지는 배포 가능한 불변 템플릿이고, 컨테이너는 그 템플릿을 실행한 격리 프로세스다. Docker의 이식성은 같은 이미지에서 오며, 빠른 실행은 운영체제 전체가 아닌 호스트 커널을 공유하는 구조에서 나온다.
