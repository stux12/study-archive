# 학습 아카이브 콘텐츠·시각자료 정책

## 왜 이 정책인가 — 이중 부호화(Dual Coding)

같은 내용을 **글 + 시각자료로 동시에** 담으면 기억·이해가 훨씬 오래간다.
이 아카이브의 목적은 "새롭게 알게 된 정보를 내 것으로 만드는 것"이므로,
**개념은 최대한 시각화**하는 것을 기본 원칙으로 삼는다.

## 1. 글 구조 (권장 틀 — 강제가 아니라, 그때그때 내용에 맞게 유연히)

정형화된 규격을 강요하지 않는다. 아래는 출발점일 뿐이며, 글의 성격에 맞게 조정한다.

- **한 줄 요약** — 이 글을 한 문장으로
- **왜 중요한가** — 배경/동기
- **핵심 개념** — 여기에 **시각자료 1개 이상**
- **예제 / 코드**
- **내가 헷갈린 점** — 오개념 기록(정교화)
- **복습 질문** — 접기 토글로 답을 가려 능동 회상(active recall)
- **참고 자료 / 관련 글**

## 2. 시각자료 정책 (핵심)

- **개념 하나당 시각자료(다이어그램·그림·표) 최소 1개**를 목표로 한다.
- 종류별 사용처:

| 시각자료 | 언제 쓰나 |
| --- | --- |
| **Mermaid 다이어그램** | 흐름·구조·순서·관계 (플로우차트·시퀀스·마인드맵·타임라인) |
| **이미지/스크린샷** | 실제 화면, 손그림, 도식 캡처 |
| **표** | 비교·정리 (개념 A vs B 등) |
| **콜아웃 박스** | 핵심 / 주의 / 팁 강조 |
| **접기 토글** | 복습 질문의 답, 곁가지·심화 내용 |
| **코드 블록** | 예제 (언어별 하이라이트) |

- **접근성**: 이미지에는 alt 텍스트를 넣는다.
- **성능**: 스크린샷은 과도하게 크지 않게(용량 최적화).

## 3. 파이프라인 지원 현황

`sync-notion.mjs`가 Notion 블록을 Markdown/HTML로 변환한다. 현재/목표 상태:

| 기능 | 상태 | 작성법(Notion) |
| --- | --- | --- |
| 문단·제목·리스트·체크박스·인용·코드·구분선·북마크 | ✅ 지원 | 기본 블록 |
| 콜아웃(아이콘·색) | 🔜 강화 예정 | Notion 콜아웃 블록 |
| 접기 토글 → `<details>` | 🔜 예정 | Notion 토글 블록 |
| 표(Notion table) | 🔜 예정 | Notion 표 블록 |
| 이미지/스크린샷 | 🔜 예정 | 이미지 붙여넣기(동기화 시 저장소로 다운로드) |
| Mermaid 다이어그램 | 🔜 예정 | 코드 블록 언어를 `mermaid`로 |
| 수식(KaTeX) | ⏸ 보류(선택) | — |

> ⚠️ **이미지 만료 주의**: Notion 이미지 URL은 약 1시간 뒤 만료된다(서명 URL).
> 그래서 동기화 시 이미지를 **저장소로 내려받아** 로컬 경로로 참조한다. Notion URL을 그대로 쓰지 않는다.

## 4. 품질·발행 규칙

- 발행 전 [검수 체크리스트](publish-review-checklist.md)에서 **시각자료 1개 이상 / alt 텍스트 / 만료 URL 없음**을 확인한다.
- `dry_run` 미리보기로 배포 대상과 경고를 먼저 본다.

## 5. 카테고리·태그 택소노미

- **카테고리**: 대분류. 글마다 **하나만** (예: `CS`, `DevOps`, `언어`, `운영 가이드`).
- **태그**: 세부 주제. 여러 개 가능. **기존 태그 재사용을 우선**해 파편화를 막는다.
- 관련 글은 본문에서 서로 링크해 지식 그물망을 만든다.
